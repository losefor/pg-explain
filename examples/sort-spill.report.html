Warning Sort spilled to disk using 180.0 MiB of temp space PGX_SORT_SPILL_DISK

What: The Sort node ran an external merge sort on disk (external merge) using 180.0 MiB of temp space, because the data exceeded work_mem.

Why: work_mem was too small to hold the sort set, so Postgres wrote sorted runs to temporary files and merged them — adding temp-file I/O that an in-memory sort avoids.

Fix: Raise work_mem for this query so the sort stays in memory, e.g. SET work_mem = '400MB' at session or role scope (do NOT raise it globally without accounting for max_connections — each connection can allocate work_mem several times over). Alternatively, add an index on (events.created_at DESC, events.user_id) so rows arrive pre-sorted and the Sort node disappears.

  • Set work_mem at session or role scope for this workload, not cluster-wide.
  • Size it above the spilled footprint (~180.0 MiB); 400MB leaves headroom.
  • Or add an index on (events.created_at DESC, events.user_id) so the sort is satisfied by an ordered scan and removed entirely.
Raise work_mem for this session
SET work_mem = '400MB';
Or set it per role
ALTER ROLE <role> SET work_mem = '400MB';
Or add an index matching the sort key
CREATE INDEX ON <table> (events.created_at DESC, events.user_id);

Note 20,000x row overestimate PGX_ROW_MISESTIMATE

What: Postgres estimated 2,000,000 rows but 100 were produced — a 20,000x overestimate.

Why: The planner's row estimate is based on statistics that are stale, missing, or too coarse for this predicate (e.g. correlated columns the planner treats as independent).

Fix: Refresh and sharpen statistics for the underlying table: run ANALYZE <relation>, raise per-column statistics targets on the predicate columns, and add extended statistics for correlated columns so the planner estimates rows correctly.

  • Refresh table statistics first; this alone often fixes the estimate.
  • If the column has a skewed/uneven distribution, raise its statistics target and re-ANALYZE.
  • If the predicate spans multiple correlated columns, create extended statistics so the planner stops assuming independence.
Refresh statistics
ANALYZE <relation>;
Raise per-column statistics target
ALTER TABLE <relation> ALTER COLUMN <column> SET STATISTICS 1000;
ANALYZE <relation>;
Add extended statistics for correlated columns
CREATE STATISTICS <stats_name> (dependencies, ndistinct) ON <col_a>, <col_b> FROM <relation>;
ANALYZE <relation>;
